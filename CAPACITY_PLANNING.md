# Team Chat 容量评估报告

## 当前架构

```
后端：Flask 3.0 + Flask-SocketIO 5.3.6
数据库：SQLite 3
文件存储：本地文件系统
实时通信：WebSocket (Socket.IO)
```

## 理论容量上限

### 1. SQLite 数据库限制

| 指标 | 理论上限 | 推荐阈值 | 当前配置 |
|------|----------|----------|----------|
| 数据库大小 | 140 TB | 10 GB | ~50 MB |

| 并发写入 | ~100 QPS | 50 QPS | - |
| 并发读取 | 无限制 | 500 QPS | - |

| 场景 | 单实例上限 | 推荐值 |

| 轻量聊天（纯文本） | 10,000 连接 | 2,000 |
| 中等负载（含文件） | 5,000 连接 | 1,000 |
| 重度负载（大文件） | 2,000 连接 | 500 |

### 3. 文件系统存储

假设配置：
- 文件保留期：10 天
- 平均文件大小：5 MB
- 每人每天上传：10 个文件

```
100 用户 × 10 文件/天 × 5 MB × 10 天 = 50 GB
1000 用户 × 10 文件/天 × 5 MB × 10 天 = 500 GB
```

### 性能瓶颈分析

#### 🔴 瓶颈 1：SQLite 并发写入

**问题**：SQLite 是文件级锁，同一时间只能有一个写入事务

**影响场景**：
- 多人同时发送消息（>50 人/秒）
- 批量导入数据
- 定时清理任务运行时

**解决方案**：
```python
# 1. 使用 WAL 模式（Write-Ahead Logging）
PRAGMA journal_mode = WAL;

# 2. 批量写入合并
# 3. 异步队列处理非实时操作
```

#### 🟡 瓶颈 2：单文件数据库

**问题**：所有数据在一个文件，备份/迁移困难

**影响场景**：
- 数据库 > 5 GB 时备份缓慢
- 磁盘故障风险集中

**解决方案**：
```python
# 1. 定期 VACUUM 优化
# 2. 分库分表（按用户/群组）
# 3. 迁移到 PostgreSQL（需要时）
```

#### 🟡 瓶颈 3：文件存储

**问题**：本地存储，无 CDN 加速

**影响场景**：
- 大文件下载慢
- 多用户同时访问同一文件

**解决方案**：
```python
# 1. 使用对象存储（阿里云 OSS/七牛云）
# 2. CDN 加速
# 3. 文件分片上传
```

## 扩容方案

### 阶段 1：当前架构优化（0-500 用户）

**成本**：¥0-500/月

```yaml
优化项:
  - SQLite WAL 模式
  - 数据库索引优化
  - 消息分页加载（当前 100 条/次）
  - 文件上传限流（100 MB/文件）
  - 定时清理（10 天自动删除）
```

### 阶段 2：中等规模升级（500-2000 用户）

**成本**：¥500-2000/月

```yaml
升级项:
  - 数据库：SQLite → PostgreSQL
  - 缓存：Redis（会话/在线状态）
  - 文件：本地 → 对象存储（OSS）
  - 部署：单机 → 负载均衡（2-4 台）
  - 监控：Prometheus + Grafana
```

### 阶段 3：大规模架构（2000-10000 用户）

**成本**：¥2000-10000/月

```yaml
升级项:
  - 数据库：PostgreSQL 集群（主从复制）
  - 缓存：Redis 集群
  - 消息队列：RabbitMQ/Kafka
  - 文件：CDN 全球加速
  - 部署：K8s 容器化（10+ 节点）
  - 监控：ELK 日志系统
```

## 结论

### 当前架构（Flask + SQLite）

**适合场景**：
- ✅ 小型团队（< 200 人）
- ✅ 内部沟通工具
- ✅ 低频率文件分享
- ✅ 预算有限（< ¥500/月）

**不适合场景**：
- ❌ 大规模并发（> 500 人同时在线）
- ❌ 高频文件传输（> 1000 文件/天）
- ❌ 企业级可靠性要求
- ❌ 全球化部署

### 建议

**短期（0-6 个月）**：
- 保持当前架构
- 实施数据库优化（WAL 模式、索引）
- 添加监控告警

**中期（6-12 个月）**：
- 用户 > 500 时迁移到 PostgreSQL
- 文件存储迁移到 OSS
- 添加 Redis 缓存

**长期（12 个月+）**：
- 根据实际增长选择云服务商
- 考虑微服务拆分
- 实施容器化部署
